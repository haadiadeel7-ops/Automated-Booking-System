---
name: Always minimize implementation steps
description: User wants the fastest possible approach — fewest files, fewest steps, no extras
type: feedback
---

Always take the shortest path. Never add steps, files, or abstractions beyond the bare minimum.

**Why:** User gets frustrated when plans/implementations are longer than necessary. They will push back every time until the approach is as minimal as possible.

**How to apply:** Before proposing any plan or change, ask "can this be done in fewer files or steps?" Default to the smallest possible change. Never pad with "nice to haves".
